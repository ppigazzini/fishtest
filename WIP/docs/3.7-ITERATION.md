# Iteration plan (Milestone 7 — Starlette idiomatic plumbing, without bloat)

Date: 2026-01-29

This iteration defines the work to make the HTTP layer idiomatic Starlette/FastAPI while preserving protocol parity and the readable entrypoint flow in `http/api.py` and `http/views.py`.

Goals
- Replace internal request‑shim patterns with typed dependencies and `Annotated` aliases where they reduce complexity.
- Adopt Starlette idioms (dependencies, middleware, sessions) without re‑introducing helper bloat.
- Keep CSRF enforcement explicit and UI 403/404 HTML behavior stable.

Metrics
- Hop count ≤ 1 per endpoint; helper calls ≤ 2 per endpoint in `http/api.py` and `http/views.py`.
- No new router trees; keep HTTP entrypoints as the primary narrative.

Constraints
- Preserve externally-visible behavior (worker JSON shapes, UI HTML errors, redirects, cookie names) at all times.
- Move incrementally; each change must be covered by contract tests from Milestone 6.
- Do not split routes into a multi‑file “router explosion.”

Phases

Phase 0 — Design & safety
- Produce a dependency map showing current `_RequestShim` usage and where `Depends` can be introduced without increasing hop count.
- Define `Annotated` aliases for common types: `RequestContextDep`, `UIContextDep`, `RunDbDep`, etc.

Reference patterns from the bloat branch (use selectively, avoid layering):
- Single shim boundary: [__fishtest-bloat/server/fishtest/web/api/internal_shim.py](__fishtest-bloat/server/fishtest/web/api/internal_shim.py)
- Template context helper (only if it reduces duplication): [__fishtest-bloat/server/fishtest/web/ui/pipeline.py](__fishtest-bloat/server/fishtest/web/ui/pipeline.py)

Phase 1 — API request plumbing (minimal shim)
- Introduce a `get_json_body()` dependency that raises controlled validation errors routed through existing handlers.
- Keep `_RequestShim` confined to the smallest boundary needed to preserve legacy method signatures.
- Keep `run_in_threadpool` boundaries; avoid changing sync handler bodies.

Phase 2 — UI dependencies (keep flow readable)
- Implement `get_ui_context(request: Request) -> UIContext` as a dependency that composes session, userdb, actiondb, and template request only when required.
- Add `CsrfDep = Annotated[None, Depends(check_csrf)]` and apply as decorator dependency for POST routes that require CSRF.

Phase 3 — Session migration (safe rollout)
- Implement Starlette session middleware while reading legacy cookies until cutover window.
- Tests: ensure `SESSION_COOKIE_NAME` remains stable; session commit/flash behavior matches legacy.
- Rollout plan: dual-read → dual-write → flip → remove legacy reader after TTL.

Phase 4 — Template boundary (optional within Milestone 7)
- Remove `TemplateRequest` only if templates are ported or an adapter is proven identical via tests.
- Otherwise keep the boundary small and explicit; avoid new helpers that obscure flow.

Acceptance
- All contract tests remain green throughout the migration.
- No increase in hop count or unreadability of HTTP entrypoints.
- Parity scripts show no unexpected diffs.

Avoid the bloat (lessons from the draft branch)
- Do not spread a single UI flow across route → helper → pipeline → context layers when the route can be readable by itself.
- Avoid duplicative dependency helpers that differ only in naming; centralize shared `Annotated` deps.
- Avoid moving control flow into middleware when explicit route logic is clearer.

Verification
- Run Milestone 6 contract tests and parity scripts after each major change.
- `uv run ruff check server/fishtest` on modified modules.
