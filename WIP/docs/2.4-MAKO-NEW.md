# New Mako templates plan (best practices, 2026)

Date: 2026-02-04

This document defines how to refactor the legacy Mako templates into a new, cleaner Mako set under `server/fishtest/templates_mako/`, aligned with modern Mako usage and the parity requirements in [WIP/docs/1-FASTAPI-REFACTOR.md](WIP/docs/1-FASTAPI-REFACTOR.md).

It complements:
- Mako catalog: [WIP/docs/2.2-MAKO.md](WIP/docs/2.2-MAKO.md)
- Jinja2 plan: [WIP/docs/2.3-JINJA2.md](WIP/docs/2.3-JINJA2.md)

## Goals

- Clean up legacy Mako templates by reducing embedded Python and clarifying structure.
- Create a new Mako template set under `server/fishtest/templates_mako/`.
- Share a common helper base with the Jinja2 migration (filters, globals, formatters).
- Preserve UI behavior and rendered output (DOM/text parity).
- Keep legacy Mako templates in-tree for upstream rebase safety.

## Status (authoritative)

- New Mako templates live in [server/fishtest/templates_mako](server/fishtest/templates_mako) and are parity-checked against legacy Mako.
- Rendering uses `TemplateLookup` with `default_filters=["h"]` and `strict_undefined=False` for parity safety.
- `MakoTemplateResponse` mirrors Starlette `TemplateResponse` behavior and emits debug metadata via request extensions.
- Shared helper base is used across renderers (aligned with Jinja2).
- Renderer selection is driven by a Python variable in the renderer module (no environment flags).

## Non-goals

- No UI redesign or layout changes.
- No changes to HTTP endpoints, session/CSRF behavior, or template data model.
- No removal of legacy Mako templates in `server/fishtest/templates/`.

## Why a new Mako set

Legacy Mako templates mix view logic with HTML and rely on Mako-specific features that make future migrations risky. A cleaner, best-practice Mako set lets us:
- Reduce embedded Python and make templates more declarative.
- Centralize formatting in shared helpers (usable by Jinja2).
- Compare output across implementations (legacy Mako vs new Mako vs Jinja2).

## Reference takeaways from the Mako upstream docs

Based on the upstream Mako docs in `___mako/doc/build/`:
- Prefer `TemplateLookup` with explicit `input_encoding` and controlled `filesystem_checks`.
- Use `default_filters` or `<%page expression_filter="...">` for consistent escaping.
- Use `<%block>` for inheritance-friendly layout; `<%def>` for reusable functions.
- Avoid `import="*"` in `<%namespace>`; prefer explicit imports for clarity and speed.
- Use `<%! %>` only for pure functions and imports (no request-specific state).
- Use `strict_undefined=True` in tests to catch missing context keys early.

## Best practices (2026)

### Template structure and inheritance

- Use a single base layout with `<%block>` sections for head/body slots.
- Keep blocks small and layout-focused; move logic into helpers.
- Prefer `<%block name="...">` over `<%def>` for layout overrides.
- Avoid `next`/`parent` unless a multi-level layout actually needs it.

### Embedded Python and helpers

- Move heavy logic out of templates into Python helpers.
- Limit `<% %>` and `<%! %>` usage to minimal glue and imports.
- Avoid per-request logic in module-level `<%! %>` blocks.
- Use macros (`<%def>`) only for simple presentation helpers.

### Escaping and filtering

- Default to HTML escaping. Use `default_filters=["h"]` or `<%page expression_filter="h">`.
- Treat `|n` (no-escape) as a strict trust boundary; document each usage.
- Map Mako filters to shared helpers (e.g., `urlencode`).

### Context discipline

- Keep template context explicit and stable; pass only what templates need.
- Always pass `request` only if the template truly needs it.
- Avoid using reserved names (`context`, `loop`, `UNDEFINED`).

### Performance and determinism

- Use `TemplateLookup` with `module_directory` for compiled template caching.
- Set `filesystem_checks=False` in production when deploys are immutable.
- Keep large loops or computed lists in Python helpers.

### Testing and parity

- Use strict undefined in tests: `strict_undefined=True`.
- Add output comparison tests (legacy Mako vs new Mako).
- Normalize HTML before diffing (whitespace folding and attribute sorting).

## Required shared helper base (Jinja2 + new Mako)

Create a single helper module for template utilities used by both renderers (name suggestion):
- `server/fishtest/http/template_helpers.py`

Responsibilities:
- URL helpers: `static_url`, `urlencode`, route helpers as needed.
- Formatting: `format_time_ago`, `format_group`, date/time display.
- Safe HTML helpers: `safe_html` or explicit `Markup` where needed.
- Convenience for shared context (e.g., `request`, `session`, `user`).

Concrete helper functions used today (migrate into the shared helper base or re-export them):
- From [server/fishtest/util.py](server/fishtest/util.py):
  - `worker_name`, `display_residual`
  - `format_bounds`, `format_results`, `is_active_sprt_ltc`
  - `format_date`, `format_time_ago`, `format_group`
  - `get_cookie`, `diff_url`, `tests_repo`
- From request/session surface (keep consistent across renderers):
  - `request.static_url(...)`
  - `request.session.get_csrf_token()` and flash helpers
- From view context (computed values passed into templates):
  - `extract_repo_from_link`

This helper base should be imported by:
- Mako renderer setup (globals/filters)
- Jinja2 renderer setup (globals/filters)

## Rendering setup (new Mako)

Add a new renderer path that mirrors the Jinja2 renderer:

- Directory: `server/fishtest/templates_mako/`
- Renderer: `server/fishtest/http/mako_new.py` (name suggestion)
- Use `TemplateLookup` with:
  - `directories=[templates_mako_dir]`
  - `input_encoding="utf-8"`
  - `default_filters=["h"]`
  - `strict_undefined=True` in tests
  - `filesystem_checks` configurable by environment
- Provide a `render_template(name, context)` function that matches the existing UI pipeline.
