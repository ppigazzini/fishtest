# Jinja2 migration plan (FastAPI/Starlette, Python 3.14)

Date: 2026-02-03

This document defines the Jinja2 replacement strategy for Mako templates, aligned with FastAPI/Starlette best practices and the parity rules in [WIP/docs/1-FASTAPI-REFACTOR.md](WIP/docs/1-FASTAPI-REFACTOR.md).

References for API details and examples live in [WIP/docs/9-JINJA2-REFERENCES.md](WIP/docs/9-JINJA2-REFERENCES.md).

## Goals

- Replace Mako templates with Jinja2 without changing externally visible UI behavior.
- Keep UI request/session/CSRF semantics identical (HTML errors, redirects, flashes).
- Preserve `request.static_url(...)` cache-busting behavior.
- Enable side-by-side rendering to validate parity during migration.

## Current refactor changes (2026-02-05)

- Jinja2 templates are in [server/fishtest/templates_jinja2](server/fishtest/templates_jinja2) and render with parity to legacy Mako.
- Shared helpers are used across renderers (for example, `tests_run_setup`).
- Jinja2 safe output is explicit (`|safe`), with HTML entities used for ASCII-safe symbols.
- Template comments that must not render are hidden with Jinja2 comments (`{# ... #}`).
- Renderer selection is driven by a Python variable in the renderer module (no environment flags).
- Metrics and scripts: see [8-TEMPLATE-METRICS.md](8-TEMPLATE-METRICS.md).

## Non-goals

- No UI redesign or markup changes unless required by syntax conversion.
- No changes to the HTTP endpoints or session/csrf logic.
- No behavior changes to worker endpoints.

## Required packages (uv)

- Add Jinja2: `uv add jinja2`
- No additional Starlette module is required to use `Jinja2Templates`.

## Required files and modules

Create or update the following files (names are suggestions; keep in `server/fishtest/http/`):

1) `server/fishtest/http/jinja.py`
- Purpose: Jinja2 environment creation and rendering helper.
- Responsibilities:
  - Build a `jinja2.Environment` with explicit autoescape for HTML.
  - Configure template loader for the new Jinja2 templates directory.
  - Register filters equivalent to Mako filters used today (safe, url encode).
  - Register global helpers needed by templates (static_url, formatters).
  - Provide a `render_template(name, context)` function to mirror Mako flow.

2) `server/fishtest/templates_jinja2/`
- Purpose: Jinja2 template directory with converted templates.
- Migration strategy: add Jinja2 equivalents one by one, keep Mako as fallback.

3) `server/fishtest/http/ui_pipeline.py` (or a small new adapter)
- Purpose: choose Mako vs Jinja2 based on feature flag or per-template map.
- Ensure session commit/forget behavior is identical regardless of renderer.

4) `server/fishtest/http/template_request.py`
- Purpose: keep the same request surface for templates (`request` object in context).
- No behavior change: the Jinja2 context should include the same object with the same methods.

5) Tests
- Add snapshot or structural tests for the hardest templates to compare Mako vs Jinja2 output.
- Add contract tests for login/logout/403/404 HTML.

## Best practices (FastAPI/Starlette, Python 3.14)

- Use explicit Jinja2 `Environment` and avoid implicit global state.
- Enable `autoescape` for HTML templates and use `|safe` only where required.
- Keep template context explicit: pass only what templates need.
- Keep template rendering off the event loop (use `run_in_threadpool` like Mako).
- Avoid coupling templates to FastAPI request object internals; keep the existing shim.
- Use deterministic escaping and avoid mixing HTML strings with unsafe input.
- Use the keyword-based `TemplateResponse(request=..., name=..., context=...)` signature.
- Always include `request` in the template context; it is required by Starlette.

## Modern 2025 practices (recommended)

- Use `jinja2.Environment` with `select_autoescape(["html", "xml"])` and avoid disabling autoescape globally.
- Keep `|safe` usage minimal and documented; treat it as an explicit trust boundary.
- Prefer explicit context objects over globals; add only a small, audited set of globals for legacy helpers.
- Use `StrictUndefined` in tests or CI snapshots to catch missing context keys early; keep production lenient if needed.
- Use a per-template allowlist for legacy `request` surface to avoid accidental template coupling to new request fields.
- Add a deterministic HTML normalization step (whitespace folding + attribute sorting) before diffing outputs.
- Use a dual-renderer strategy until parity is proven, then cut over template-by-template.

## Template context contract

Keep the same context surface used by Mako:
- `request.session.get_csrf_token()`, `request.session.flash()/pop_flash()`
- `request.static_url("fishtest:static/..." )`
- `request.cookies`, `request.headers`, `request.authenticated_userid`
- Any computed helpers currently passed into template context

## Filters and globals mapping

- Mako `|h` -> Jinja2 `|e`
- Mako `|n` -> Jinja2 `|safe`
- Mako `|u` -> Jinja2 `|urlencode`

Add Jinja2 filters:
- `urlencode`: from `urllib.parse.quote_plus` or `quote`
- `format_time_ago`, `diff_url`, `get_cookie`, `format_group`, etc. as globals

## Rendering flow (parity-preserving)

- Keep the render call inside the same threadpool boundary as Mako.
- Preserve response headers and cookie commit logic in `boundary.py` and `ui_pipeline.py`.
- Use the same template name mapping to avoid route changes.
- Feature flag should choose renderer based on:
  - A per-template map (preferred) or
  - A global flag for a full cutover

## DOM diff strategy and acceptance thresholds

- Normalize output before comparison:
  - Collapse runs of whitespace.
  - Remove insignificant whitespace between tags.
  - Sort attributes in a stable order for diffing.
- Compare both raw HTML and normalized DOM output for each converted template.
- Acceptance thresholds:
  - Zero changes in text content (except whitespace-only diffs).
  - Stable element IDs, classes, and data attributes.
  - Stable form field names and hidden input values (CSRF, ids).
  - Stable link hrefs and query parameter ordering.
- Any functional or text diffs require an explicit, documented approval.

## Conversion phases

Phase 0 — Infrastructure
- Add Jinja2 package via `uv add jinja2`.
- Create Jinja2 environment (`http/jinja.py`).
- Add a `render_jinja_template` helper and integrate into UI pipeline.
- Add a feature flag and per-template switch map.

Phase 1 — Base and shared partials
- Convert `base.mak`, `pagination.mak`, `run_tables.mak`, `run_table.mak` to Jinja2.
- Verify template inheritance and includes work correctly.
- Add snapshot tests for base layout and run table blocks.

Phase 2 — Hardest-first conversions
- Convert templates in the order defined in [WIP/docs/3.8-ITERATION.md](WIP/docs/3.8-ITERATION.md).
- For each template:
  - Replace Mako control flow with Jinja tags.
  - Move module-level Python into Python helpers or Jinja globals.
  - Preserve string formatting and spacing for user-visible fields.
  - Update `include` and `macro` usage as needed.
  - Run snapshot tests and UI flow tests.

Phase 3 — Cleanup and cutover
- Remove unused Mako helpers and Mako environment.
- Make Jinja2 the default renderer.
- Keep contract tests and parity scripts green.

## Required template conversions (per-file mapping)

- `base.mak` -> `base.html` with `{% block head %}` and `{{ self.body() }}` mapped to `{% block body %}`.
- `pagination.mak` -> `pagination.html` using Jinja `for` and `if`.
- `run_table.mak` and `run_tables.mak` -> `run_table.html` and `run_tables.html` with macros for reuse.
- `elo_results.mak` -> `elo_results.html`, moving complex Python into helper functions.
- `tests_view.mak`, `tests_stats.mak`, `tests_run.mak` -> move heavy Python to helpers and keep templates mostly declarative.

## Safety checks

- Compare rendered HTML for each converted template against Mako output (snapshot or DOM diff).
- Verify CSRF token presence in forms.
- Verify flash messages still render correctly.
- Verify error pages remain HTML and not JSON.
- Verify static_url cache-busting remains stable.

## Deliverables checklist

- Jinja2 environment module in `server/fishtest/http/jinja.py`.
- Jinja2 template directory `server/fishtest/templates_jinja2/`.
- Updated UI render path to select Jinja2 per template.
- Snapshot tests for at least the top 4 highest-complexity templates.
- Documentation updates in [WIP/docs/3.8-ITERATION.md](WIP/docs/3.8-ITERATION.md).
