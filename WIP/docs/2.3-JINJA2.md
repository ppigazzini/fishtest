# Jinja2 migration plan (FastAPI/Starlette, Python 3.14)

Date: 2026-02-03

This document defines the Jinja2 replacement strategy for Mako templates, aligned with FastAPI/Starlette best practices and the parity rules in [WIP/docs/1-FASTAPI-REFACTOR.md](WIP/docs/1-FASTAPI-REFACTOR.md).

References for API details and examples live in [WIP/docs/9-JINJA2-REFERENCES.md](WIP/docs/9-JINJA2-REFERENCES.md).

## Goals

- Replace Mako templates with Jinja2 without changing externally visible UI behavior.
- Keep UI request/session/CSRF semantics identical (HTML errors, redirects, flashes).
- Preserve `request.static_url(...)` cache-busting behavior.
- Enable side-by-side rendering to validate parity during migration.

## Status (authoritative)

- Jinja2 templates live in [server/fishtest/templates_jinja2](server/fishtest/templates_jinja2) and are parity-checked against legacy Mako.
- Jinja2 rendering uses Starlette `Jinja2Templates` with a custom `Environment` (see [server/fishtest/http/jinja.py](server/fishtest/http/jinja.py)).
- Autoescape is enabled for `html`, `xml`, and `mak` extensions to align with new Mako defaults.
- `MakoUndefined` is used to mimic legacy `UNDEFINED` rendering behavior.
- Shared helpers from [server/fishtest/http/template_helpers.py](server/fishtest/http/template_helpers.py) are registered as filters/globals.
- Rendering stays synchronous and is invoked off the event loop; response metadata is attached by the response adapter.

## Non-goals

- No UI redesign or markup changes unless required by syntax conversion.
- No changes to the HTTP endpoints or session/csrf logic.
- No behavior changes to worker endpoints.

## Dependencies

- `jinja2` must be installed (Starlette templates treat it as optional).
- No additional Starlette module is required to use `Jinja2Templates`.

## Implementation

- Renderer: [server/fishtest/http/jinja.py](server/fishtest/http/jinja.py)
- Templates: [server/fishtest/templates_jinja2](server/fishtest/templates_jinja2)
- Template request shim: [server/fishtest/http/template_request.py](server/fishtest/http/template_request.py)

## Best practices (FastAPI/Starlette, Python 3.14)

- Use explicit Jinja2 `Environment` and avoid implicit global state.
- Enable `autoescape` for HTML templates and use `|safe` only where required.
- Keep template context explicit: pass only what templates need.
- Keep template rendering off the event loop (use `run_in_threadpool` like Mako).
- Avoid coupling templates to FastAPI request object internals; keep the existing shim.
- Use deterministic escaping and avoid mixing HTML strings with unsafe input.
- Use the keyword-based `TemplateResponse(request=..., name=..., context=...)` signature.
- Always include `request` in the template context; it is required by Starlette.

## Idiomatic Jinja2 pattern (repo-specific, 2026)

High-level pattern to follow for every template:
- Views build a **typed, explicit context object** with preformatted strings and URLs.
- Templates are mostly declarative: they render `*_label` strings and `*_url` links.
- Use `extends` + `block` for layout, and `include`/`macro` for shared UI fragments.
- The only request-adjacent behavior in templates is `url_for`/`static_url` helpers.
- JS data is passed as JSON via `|tojson`, not string interpolation.
- Any HTML marked safe is already sanitized or hardcoded by helpers.

## Jinja2 idiomatic patterns checklist (repo)

Macros and includes:
- [ ] Reusable fragments are implemented as macros or includes, not copy/paste.
- [ ] Macro arguments are plain values (no request access).
- [ ] Includes use `with context` only when explicitly required.

Blocks and layout:
- [ ] Each page defines `block title` and `block body` only; avoid inline `document.title`.
- [ ] Layout-only logic lives in base templates; page logic lives in page templates.
- [ ] `block head` is used only for page-specific assets.

Helpers and context:
- [ ] No template queries DB or request internals; views pass explicit context.
- [ ] URLs are built with `url_for` or prebuilt `*_url` strings.
- [ ] Dates and numbers are preformatted in views (`*_label`).
- [ ] JS data uses `|tojson` and does not embed raw strings.
- [ ] `|safe` only appears on values produced by audited helpers.

## Practices (current)

- Use `jinja2.Environment` with `select_autoescape(["html", "xml", "mak"])` and avoid disabling autoescape globally.
- Keep `|safe` usage minimal and documented; treat it as an explicit trust boundary.
- Prefer explicit context objects over globals; add only a small, audited set of globals for legacy helpers.
- Use `StrictUndefined` in tests or CI snapshots to catch missing context keys early; keep production lenient if needed.
- Use a per-template allowlist for legacy `request` surface to avoid accidental template coupling to new request fields.
- Add a deterministic HTML normalization step (whitespace folding + attribute sorting) before diffing outputs.
- Use a dual-renderer strategy until parity is proven, then cut over template-by-template.

## Template context contract

Keep the same context surface used by Mako:
- `request.session.get_csrf_token()`, `request.session.flash()/pop_flash()`
- `request.static_url("fishtest:static/..." )`
- `request.cookies`, `request.headers`, `request.authenticated_userid`
- Any computed helpers currently passed into template context

## Filters and globals mapping

- Mako `|h` -> Jinja2 `|e`
- Mako `|n` -> Jinja2 `|safe`
- Mako `|u` -> Jinja2 `|urlencode`

Add Jinja2 filters:
- `urlencode`: from `urllib.parse.quote_plus` or `quote`
- `format_time_ago`, `diff_url`, `get_cookie`, `format_group`, etc. as globals

## Rendering flow (parity-preserving)

- Keep the render call inside the same threadpool boundary as Mako.
- Preserve response headers and cookie commit logic in `boundary.py` and `ui_pipeline.py`.
- Use the same template name mapping to avoid route changes.
- Feature flag should choose renderer based on:
  - A per-template map (preferred) or
  - A global flag for a full cutover

## DOM diff strategy and acceptance thresholds

- Normalize output before comparison:
  - Collapse runs of whitespace.
  - Remove insignificant whitespace between tags.
  - Sort attributes in a stable order for diffing.
- Compare both raw HTML and normalized DOM output for each converted template.
- Acceptance thresholds:
  - Zero changes in text content (except whitespace-only diffs).
  - Stable element IDs, classes, and data attributes.
  - Stable form field names and hidden input values (CSRF, ids).
  - Stable link hrefs and query parameter ordering.
- Any functional or text diffs require an explicit, documented approval.

## Safety checks (required)

- Compare rendered HTML for each template against Mako output (DOM-normalized parity).
- Verify CSRF token presence in forms.
- Verify flash messages still render correctly.
- Verify error pages remain HTML and not JSON.
- Verify static_url cache-busting remains stable.
