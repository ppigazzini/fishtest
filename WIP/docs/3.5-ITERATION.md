# Iteration plan (Milestone 5 — remove Pyramid wrapper scaffolding, keep protocol parity)

Date: 2026-01-26

This doc is the Milestone 5 work plan: stop “pretending Pyramid” in the serving layer by removing the Pyramid-style wrapper patterns
(request shims + view-config dispatch) and replacing them with explicit, idiomatic FastAPI handlers and dependencies, while preserving
Pyramid-era **behavior contracts** (worker protocol + UI protocol).

## North star (Milestone 5)

Make the server “FastAPI-native” at the route layer:

- UI and API endpoints are explicit FastAPI route handlers (`@router.get`, `@router.post`, `add_api_route`).
- Dependencies (`Depends(...)`) become the way we access request-scoped state (db handles, user identity, CSRF/session primitives).
- The Pyramid wrapper scaffolding is deleted:
  - no `__view_configs__`-style registry
  - no generic `_dispatch_view(...)` trampoline
  - no Pyramid request object as the internal programming model

At the same time:

- Worker protocol and UI behavior contracts stay stable; intentional behavior changes must be explicit and test-gated.

Non-goals (Milestone 5):

- No UI redesign.
- No “make everything async” rewrite.
- No template engine migration (see Milestone N-1).

## Definition of done (Milestone 5)

1) **Handlers**

- UI and API surfaces are expressed as idiomatic FastAPI handlers with typed inputs and `Depends(...)`.
- Route wiring does not rely on Pyramid-style decorators/metadata to locate a view.

2) **Scaffolding removed**

- No Pyramid-style request-shim-as-primary-programming-model.
- No view-config dispatch/registry layer required to serve requests.

3) **Boundary-only compatibility**

- Any Pyramid-compat objects exist only at true boundaries (e.g., minimal template request object while Mako templates remain).

4) **Parity gates remain**

- Contract tests and parity scripts remain the safety net; they stay green.

## Constraints and guardrails

### Hotspots (minimize churn)

Avoid broad edits and formatting-only churn in:

- [server/fishtest/glue/views.py](../../server/fishtest/glue/views.py)
- [server/fishtest/glue/api.py](../../server/fishtest/glue/api.py)

Use the two-step landing strategy from [3.0-ITERATION-RULES.md](3.0-ITERATION-RULES.md):

1) helpers-only commit (additive)
2) minimal call-site swap (surgical)

### Protocol safety invariants (stop-the-line)

Pause and fix before proceeding if any of these regress:

- Worker responses change shape or lose `duration` (especially on error paths).
- UI errors start returning JSON (404/403/CSRF/etc).
- Login persistence, flashes, or CSRF semantics drift.
- Blocking work moves onto the event loop.

## Milestone 5 work plan (phases / tasks)

Land each numbered item as its own PR/commit where feasible.

### Phase 0 — Inventory + target shape (make the deletion plan concrete)

Goal: agree on what we are deleting and what replaces it.

1) Write down the “wrapper scaffolding” inventory and deletion targets (code pointers in the PR description):
   - view registration/dispatch (what currently maps URL → view function)
   - request shim surface (what attributes the legacy code expects)
   - template boundary (what Mako templates need)

2) Define the target handler/dependency surface for both UI and API:
   - dependencies for: current user identity, session access, CSRF, db handles, base URL
   - handler signature patterns (typed query/form payloads, explicit defaults)

Acceptance:

- A reviewer can answer “what gets deleted” and “what replaces it” without reading the whole repo.

### Phase 1 — Introduce explicit FastAPI routers (no behavior change)

Goal: route wiring becomes explicit, without changing view logic yet.

3) Create/standardize routers:
   - one router for UI endpoints
   - one router for API endpoints

4) For each route, define a minimal handler that:
   - parses inputs explicitly (query params / form)
   - obtains dependencies via `Depends(...)`
   - calls the existing implementation (initially still in legacy modules), using a minimal boundary object if required

Constraint:

- This is a wiring-only change: behavior and templates remain identical.

Acceptance:

- You can locate every externally served endpoint by reading the routers.

### Phase 2 — Shrink request shims (boundary-only compatibility)

Goal: stop using a Pyramid-like request object as the *internal* programming model.

5) Introduce an explicit “UI context” object (or equivalent) constructed by dependencies:
   - identity (authenticated userid)
   - permissions/roles flags
   - db handles
   - URL helpers
   - template boundary object (only what templates truly require)

6) Update a small number of low-risk, read-only UI routes to use the new context directly.

7) Iterate route-by-route, keeping diffs small.

Acceptance:

- New/converted handlers do not require the full request shim.
- Any remaining compatibility request object is template-boundary-only.

### Phase 3 — Delete view-config dispatch/registry layer

Goal: remove the trampoline/registry patterns entirely.

8) Remove `__view_configs__`-driven lookup / generic dispatch from the serving path.

9) Ensure permission/CSRF/session behavior is handled by explicit dependencies/middleware.

Acceptance:

- There is no view registry + dispatcher needed to serve requests.

### Phase 4 — Reduce Pyramid test stubs dependency (optional within Milestone 5)

Goal: tests validate the active server behavior, not Pyramid stubs.

10) Prefer FastAPI HTTP tests for active endpoints where practical.

11) Keep Pyramid stubs only if they still cover important pure-function logic, and ensure they don’t block milestone goals.

Acceptance:

- The contract-test gate meaningfully exercises the FastAPI handlers.

## Verification gates (minimum bar)

From `server/`:

- `uv run python -m unittest discover -s tests -q`

Parity scripts (repo root), when the change touches route surfaces or glue bodies:

- `uv run python WIP/tools/parity_check_api_routes.py`
- `uv run python WIP/tools/parity_check_views_routes.py`
- `uv run python WIP/tools/parity_check_api_ast.py` (when touching API glue bodies)
- `uv run python WIP/tools/parity_check_views_ast.py` (when touching UI glue bodies)

## Notes for reviewers

- Expect a sequence of small PRs rather than a flag-day rewrite.
- “Idiomatic FastAPI” is a tool for clarity/testability; it is not a style exercise.
- Any behavior change must be called out explicitly, paired with a contract test, and reviewed as a protocol change.
