# Iteration plan (Milestone 10 — Jinja2 idiomatic cutover, retain legacy Mako)

Date: 2026-02-07
Status: Planned

## Goals
- Produce idiomatic Jinja2 templates for Starlette in [server/fishtest/templates_jinja2](server/fishtest/templates_jinja2).
- Preserve legacy Mako in [server/fishtest/templates](server/fishtest/templates) for rebase safety.
- Keep parity tooling for legacy Mako vs the parity Jinja2 set (moved to a temporary path).
- Retire the new Mako templates and renderer that were created only for parity work.

## Requirements
- Legacy templates remain read-only and runnable for upstream rebase safety.
- Rendering stays off the event loop (threadpool boundary preserved).
- `TemplateResponse(request=..., name=..., context=...)` remains the UI return path.
- The new Jinja2 templates are not required to be line-for-line comparable with Mako.
- Parity scripts remain valid for legacy Mako vs the parity Jinja2 set.
- Avoid large refactors in `http/api.py` and `http/views.py`; only small signature/context changes that preserve parity tracking are allowed.
- ASCII-only source where practical; use HTML entities when needed.

## Phases

### Phase 0 — Inventory and cutover map
Checklist:
- [ ] List every template and its legacy Mako counterpart.
- [ ] Identify templates that rely on request-side effects or helper globals.
- [ ] Record templates with heavy logic or unsafe HTML insertion.
- [ ] Validate the per-template context contract list in [WIP/docs/10.0-JINJA2-CONTEXT-CONTRACTS.md](WIP/docs/10.0-JINJA2-CONTEXT-CONTRACTS.md).

### Phase 1 — Split parity Jinja2 from idiomatic Jinja2
Checklist:
- [ ] Move the current parity Jinja2 set to `templates_jinja2_tmp`.
- [ ] Update parity scripts to point at the temporary Jinja2 path.
- [ ] Keep legacy Mako comparisons green for the temporary set.

### Phase 2 — Build idiomatic Jinja2 templates
Checklist:
- [ ] Create new Jinja2 templates in [server/fishtest/templates_jinja2](server/fishtest/templates_jinja2).
- [ ] Follow the per-template context contracts in [WIP/docs/10.0-JINJA2-CONTEXT-CONTRACTS.md](WIP/docs/10.0-JINJA2-CONTEXT-CONTRACTS.md).
- [ ] Use modern Jinja2 structure (inheritance, macros, blocks).
- [ ] Prefer view-provided context over `request` access in templates.
- [ ] Use `tojson` for JS data and avoid inline string concatenation in JS.

### Phase 3 — Minimal API/view adjustments (parity-safe)
Checklist:
- [ ] Add small context-shaping helpers where needed (no large refactors).
- [ ] Keep signatures and behavior comparable to legacy twins for parity tracking.
- [ ] If any changes touch `http/api.py` or `http/views.py`, keep them surgical and paired with parity scripts.

### Phase 4 — Retire new Mako templates and renderer
Checklist:
- [ ] Remove or disable `templates_mako` and any new-Mako-specific renderer wiring.
- [ ] Keep legacy Mako rendering path intact for upstream rebase safety.

## Deliverables
- `templates_jinja2_tmp` holds the parity Jinja2 set used by scripts.
- A new idiomatic Jinja2 template set exists in `templates_jinja2` and renders correctly in the UI.
- Parity tooling continues to validate legacy Mako vs the temporary Jinja2 set.
- Documentation updated to reflect the split and the new Jinja2 goals.
